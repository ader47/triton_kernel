# H2D 与通信冲突分析

## 背景

### KV offload

![](./58fc3d49-7f6a-48ba-b951-f3d646577eb7.png)

KV offload 场景下，后端会在推理过程中持续把 KV cache 从 Host 侧搬运到 NPU 侧。该 H2D 传输与模型前向计算、HCCL 集合通信以及运行时算子下发存在时间重叠，因此需要同时关注数据面和控制面两类资源竞争。

### 现象

Profiling 中可以观察到 HCCL 通信出现明显“快慢卡”：部分卡由于算子下发较晚，通信算子启动时间落后；集合通信需要所有参与卡同时进入通信阶段，因此先启动的卡会等待后启动的卡，最终表现为通信耗时被拉长。

如下图所示，上方算子下发过程较慢，下方算子下发较快。下发快的通信算子已经到达等待点，但需要等待下发慢的卡进入同一个通信阶段。

![image](./3009faab-13db-4cb7-aa63-038c884d3843.png)

## 结论概览

当前实验更支持如下判断：

1. A3 上的主要冲突点不在 CPU 调度，也不在 Stars 单一队列堆积，而更可能位于 H2D、HCCL 集合通信产生的 D2D 流量、算子下发共同经过的 HCCS 共享传输路径上。
2. 参与 H2D 的 NPU 数量从 16 减少到 4 或 2 后冲突消失或显著减弱，这是定位共享路径瓶颈的关键证据：如果问题主要来自单卡 SDMA/MTE 引擎，减少其他卡的 H2D 并发通常不应显著改善。
3. H2D 会同时影响两类流量：
   - 数据面：大粒度 H2D DATA 与 HCCL 集合通信产生的 D2D DATA 共享链路或交换资源，导致 D2D 通信可插入的调度空隙减少。
   - 控制面：高并发 H2D REQ 会与算子下发、HCCL 通信控制请求等 REQ 流量竞争，导致动态单算子模式下的算子下发延迟增加。
4. 模型图模式可以显著缓解控制面问题，因为大量算子下发和依赖关系被静态化或批量化，运行时不再依赖高频单算子动态下发。但如果大包 H2D DATA 仍与 HCCL 集合通信产生的 D2D DATA 时间重叠，数据面冲突仍可能存在。
5. QoS 优先级主要能缓解 REQ 类控制流量竞争；当 DATA 流量无法按业务类型有效区分优先级时，需要通过拆小 H2D 粒度、错峰、限流或改变分发方式来减少大包阻塞。

## 分析过程

### 第一阶段：排除 CPU 绑核问题

**现象**：A3 上观测到 HCCL 通信快慢卡。慢卡并不是通信算子本身一定更慢，而是通信算子启动更晚，导致其他卡在集合通信入口处等待。

**排查动作**：

- 使用 cgroup 给 memcache 后端绑核，隔离 memcache 后端和 vLLM 框架使用的 CPU 核。
- 使用 `"enable_cpu_binding": "true"` 给 vllm-ascend 框架绑核。
- 目标是排除 CPU 调度抖动、运行时线程抢占、CPU 核混用带来的算子下发延迟。

**结果**：绑核后问题仍然存在。

**推论**：CPU 调度不是 A3 问题的主因。CPU 抖动可能会放大动态下发延迟，但无法解释绑核后仍稳定存在的 H2D 与通信冲突。

### 第二阶段：排查 Stars 引擎队列堆积

普通算子和部分 H2D 任务都可能经过 Stars 相关路径下发，因此初始怀疑是 Stars 任务队列堆积：大 H2D 任务阻塞后续小通信任务或算子控制任务，最终导致算子启动晚、通信快慢卡。

![](./efeb06c8-8d32-4a9d-b4c9-4adda682fda9.png)

**假设**：Stars 引擎任务队列排队导致通信任务或算子下发被 H2D 阻塞。

**验证**：将 H2D 改为通过 MTE（Memory Transfer Engine）算子方式下发，尽量绕开原先怀疑的 Stars 队列路径。

**结果**：

- 通信仍然劣化，说明冲突并不只发生在 Stars 队列。
- 计算也出现劣化。MTE 算子方式会占用计算侧资源，与前向计算算子争抢 AIV 等执行资源；即使尝试给 H2D 和计算算子分配不同数量的 AIV 核，计算仍会受影响。

![image](./17bf4b23-40be-4364-9bed-83935f5d04c5.png)

![image](./c9a6a413-5dc7-49b8-8c31-28aee6e5c30f.png)

**推论**：

- 冲突不是 Stars 队列堆积这一单点问题。
- MTE 算子方案会把原本的数据搬运问题转化为计算资源竞争问题，因此不是理想解法。
- 问题更可能出现在算子调度之下的共享硬件传输路径或交换资源。

- 这说明瓶颈主要不是“任务数太多”，而是“总 H2D 数据量及其占用共享传输资源的时间太长”。

#### 假设 2：H2D 使用 stream 过多

memcache 传输默认使用 32 个 stream。每传完一层后切换 stream 继续传下一层，可能带来 CPU 侧 stream 切换开销或运行时调度开销。memcache 聚合后，同一次接口调用内通常只使用一个 stream，不同调用之间会使用不同 stream。

**操作**：减少并发 stream 数量。

**结果**：无明显改善。

**推论**：stream 数量可能影响调度抖动，但不是当前 A3 冲突的主因。

### 第四阶段：H2D 与 D2H 对比

**现象**：D2H 不产生明显冲突，H2D 会产生明显冲突。

**解释**：

- H2D 是 Host 到多个 NPU 的数据分发，容易在 Host 入口、交换节点、共享通道或内部 buffer 上形成扇出压力。
- D2H 是 NPU 到 Host 的回传方向，流量方向、排队位置、与动态算子下发路径的重叠方式都不同，因此不一定会触发同样的队列阻塞。
- 因此，H2D/D2H 差异说明问题不只是“搬运数据就慢”，而是 H2D 方向的流量模式更容易与前向过程中的关键路径发生竞争。

**推论**：H2D 与 D2H 的差异更支持“方向相关的链路占用或排队方式”这一判断，但这一阶段还不能直接推出具体是哪类报文或哪条队列在竞争，需要结合后续实验和流量模型继续收敛。

### 第五阶段：A3 与 A2 对比实验

并行 CPU 操作也可能影响前向过程。例如 memcache 后端在做 block 查询、聚合、元数据维护、Host 侧内存拷贝、stream/事件管理时，会与推理框架的运行时线程竞争 CPU 核、cache、内存带宽或 driver/runtime 内部锁。对于动态单算子模式，这类 CPU 侧抖动会直接放大算子下发延迟。

因此需要区分两类问题：

- CPU 侧竞争：可通过绑核、隔离后端线程、降低 Host 侧并发等方式缓解。
- HCCS/链路侧竞争：即使去掉并行 CPU 操作仍存在，需要从传输路径、QoS、流量粒度和并发时序上处理。

| 实验 | A3（HCCS） | A2（HCCS） | A2（RDMA） | 观察 |
| ---- | ---------- | ---------- | ---------- | ---- |
| 默认 H2D 并行 | 劣化 | 劣化 | 劣化 | 三种场景都出现劣化 |
| 去掉并行 CPU 操作 | 劣化 | 劣化 | 不劣化 | A2 RDMA 恢复；A3/A2 HCCS 仍劣化 |

**阶段结论**：

- A2 RDMA 场景在去掉并行 CPU 操作后恢复，说明该场景中的劣化与并行 CPU 操作相关性更强。
- A3 HCCS 和 A2 HCCS 在去掉并行 CPU 操作后仍劣化，说明 HCCS 场景里还存在 CPU 操作之外的竞争因素。
- 所以这里不能简单说“A2 是 CPU 问题、A3 是链路问题”。更直接地看，A2 RDMA 这组更像是被并行 CPU 操作拖慢；而 A3 HCCS 和 A2 HCCS 即使去掉并行 CPU 操作仍然变慢，说明问题还要继续看 HCCS 传输路径。

### 第六阶段：模型图模式与单算子模式对比

在单算子动态下发模式下，H2D 不仅会劣化 HCCL 通信，还会导致算子下发延迟增加，表现为部分 NPU 因算子未及时启动而成为慢卡。

切换到模型图模式后，算子下发相关问题显著缓解。这里的图模式指的是模型执行图模式，而不是 H2D 传输本身的图模式。开启模型图模式后，模型算子、依赖关系和执行序列被提前组织，运行时模型算子下发数量显著减少，因此 H2D 对动态算子下发路径的干扰也随之减弱。

| 模式 | 模型算子下发特征 | 冲突表现 |
| ---- | ---------------- | -------- |
| 单算子模式 | 模型算子在运行时逐个动态下发，下发数量多、频率高 | H2D 与模型算子下发、HCCL 通信重叠时，容易出现慢卡和通信等待 |
| 模型图模式（A3） | 模型算子下发被静态化或批量化，运行时下发数量显著减少 | 算子下发冲突显著缓解；若大包 H2D 仍与 HCCL 集合通信产生的 D2D 流量重叠，通信侧冲突仍存在 |

**结论**：模型图模式主要缓解算子动态下发过多导致的控制面冲突，但没有消除 H2D 数据传输与 HCCL 通信之间的冲突。这说明通信侧仍存在独立于算子下发路径的共享资源竞争，需要在后续实验中继续定位。

### 第七阶段：优先级实验

**操作**：在独立实验中调整任务或通信相关流量的优先级，使延迟敏感的算子下发、HCCL 通信控制请求优先于 H2D 相关请求。

**结果**：快慢卡现象有所缓解，但没有完全恢复到无 H2D 的基线水平。

**推论**：

- 优先级调整有效，说明冲突中存在一部分可被调度策略缓解的控制面竞争。
- 优先级不能根除问题，说明仍存在无法仅靠优先级解决的共享资源占用或链路排队。
- 结合后续流量模型看，残留问题更可能来自大粒度 H2D 数据传输对共享通道的长时间占用。

### 第八阶段：设备数量实验

**操作**：减少参与 H2D 的 NPU 数量，例如从 16 张卡减少到 4 张或 2 张。

**结果**：冲突消失或显著减弱，具体表现取决于参与设备数和 H2D 并发强度。

**推理**：

- 如果瓶颈是单 NPU 的 SDMA/MTE 引擎，减少其他 NPU 的 H2D 不应显著改善单卡上的冲突。
- 减少参与设备数后冲突明显缓解，说明瓶颈更可能出现在多 NPU 共享的交换路径、虚通道、缓存队列或 Host 侧注入路径上。
- 这个现象更像是 Host 向多张 NPU 分发数据时，在共享交换节点上形成了排队或拥塞；如果多路流量又在同一个交换节点或虚通道汇聚，也可能进一步放大等待。

**进一步思考**：对于多卡读取相同 KV 数据的场景，如果每张卡都从 Host 读取一份相同数据，会造成重复 H2D 分发。潜在优化方向是“Host 只读一次，再由设备侧或交换侧广播/转发”。但该方案需要确认硬件和运行时是否支持高效广播/组播，是否能设置对应 QoS，以及额外 D2D 转发是否会反过来挤占 HCCL 通信。

## 流量模型

下面的图可以把实际传输过程简化理解为：CPU 与 NPU、NPU 与 NPU 的控制请求和数据传输都需要经过 HCCS 交换路径。不同业务流量如果映射到相同虚通道或共享 buffer，就会互相影响。

### 流量类型

| 流量 | 作用 | 典型来源 | 特点 |
| ---- | ---- | -------- | ---- |
| REQ | 控制请求、任务描述、传输描述符 | 算子下发、H2D、HCCL 通信控制请求 | 小包、延迟敏感，通常更适合做 QoS 优先级控制 |
| DATA | 真实 payload 数据 | H2D 数据、D2D 通信数据 | 大包、占带宽；如果无法按业务打 QoS 标签，则难以区分 H2D DATA 和 HCCL 集合通信产生的 D2D DATA |
| RSP | 响应或完成通知 | 设备/交换路径返回 | 通常数据量较小，但可能影响控制闭环 |
| NG | 平台定义的其他控制类流量 | 依赖具体硬件/驱动定义 | 本文不展开，仅保留分类 |

### 算子下发

算子下发主要是控制面流量。CPU 侧 runtime/driver 生成任务请求，通过 HCCS 路径发送到目标 NPU。它的数据量不大，但对延迟非常敏感；动态单算子模式下，下发慢会直接导致通信算子启动晚。

![image](./ac5efd19-a554-4f57-a514-2c95961d97c3.png)

### H2D 传输

H2D 同时包含控制请求和真实数据：

- REQ：描述传输任务、目标地址、长度、依赖关系等。
- DATA：Host 到 NPU 的实际 KV cache 数据。

当多个 NPU 同时从 Host 读取数据时，H2D DATA 会在共享链路或交换节点上形成持续带宽压力。若每张卡读取相同数据，还会产生重复分发。

![image](./bfb473cd-2ec0-4a9f-ae24-4ce9c2fe2552.png)

### HCCL/D2D 通信

HCCL 是上层集合通信库，实际在硬件链路上会产生 NPU 间 D2D 流量。D2D 通信同样包含 REQ 和 DATA。HCCL 集合通信还具有同步属性：只要某一张卡进入通信晚，其他卡即使数据通路空闲，也可能在集合通信入口或内部同步点等待。

![image](./4d6934a3-3b39-470b-98d5-4da4ed8db2d3.png)

### 交换机拓扑

![](./e0749721-e67b-4179-8734-3e88af92d69e.png)

如上图所示，CPU 与 NPU 之间通信、NPU 与 NPU 之间通信都会经过交换节点。交换机可以根据报文中的 QoS 标签进行重映射，将不同 QoS 映射到不同虚通道（VL），再通过虚通道优先级实现差异化调度。

需要注意的是：QoS 对“能识别、能打标、能映射”的流量才有效。如果 DATA 流量不带 QoS 标签，或者 H2D DATA 与 D2D DATA 在硬件上无法区分，就无法仅靠优先级让 HCCL 集合通信产生的 D2D DATA 抢占 H2D DATA。

## 冲突点分析

### 冲突点 1：DATA 大包占用共享通道

在本次测试环境中，DATA 流量无法按业务类型设置有效 QoS。H2D DATA 和 D2D DATA 如果进入同一个通道或共享同一类 buffer，就只能按硬件调度策略排队。

当 H2D 数据量很大、单次传输粒度也很大时，HCCL 集合通信产生的 D2D DATA 可插入的空隙变少，容易出现 head-of-line blocking。表现到上层就是通信启动或通信推进被拖慢。

把 H2D 拆成更小的传输粒度后，D2D DATA 可以在 H2D 小包之间穿插，等待时间缩短，冲突减轻。但代价是 H2D 传输效率可能下降，包括描述符开销增加、带宽利用率下降、调度开销上升等。因此拆包粒度需要在“通信延迟”和“H2D 带宽”之间调参。

![image](./e2ea98d5-b12b-4403-b096-7037c6af07be.png)

### 冲突点 2：REQ 拥塞影响算子下发

H2D 并发较高时，会产生大量 REQ 请求进入交换路径。这些 REQ 可能与算子下发、HCCL 通信控制请求共享队列或虚通道，导致动态单算子模式下算子下发变慢。

该问题可以通过 QoS 优先级缓解：

- 算子下发 REQ：高优先级。
- HCCL 通信控制 REQ：高优先级。
- H2D REQ：低优先级。

这样可以优先保证延迟敏感的控制流量，减少慢卡现象。但该方法主要处理 REQ 类控制流量，对 DATA 大包占用共享通道的问题帮助有限。

## 优化建议

1. **优先使用图模式或静态化执行路径**：减少运行时单算子动态下发，规避 H2D REQ 对控制面的干扰。
2. **设置 REQ 优先级**：将算子下发和 HCCL 通信控制请求优先级设置高于 H2D REQ，缓解非图模式下的快慢卡。
3. **拆分 H2D 大包**：将大粒度 H2D DATA 拆成适中的小包，让 HCCL 集合通信产生的 D2D DATA 有插入机会。需要通过实验寻找带宽损失和通信延迟改善之间的平衡点。
4. **限制 H2D 并发或错峰调度**：减少同时参与 H2D 的设备数、stream 数或层数，尽量避开 HCCL 高频通信窗口。
5. **继续保留 CPU 隔离**：虽然 A3 上的主要冲突不在 CPU 调度，但 CPU 操作会影响动态下发延迟；A2 RDMA 场景尤其需要关注 CPU 侧竞争。
6. **验证 Host 一次读取 + 设备侧广播方案**：如果多卡读取相同数据，可以评估是否支持 Host 只搬一份，再通过 NPU 间 broadcast/multicast 分发。该方案的关键验证点包括硬件支持、QoS 可控性、额外 D2D 对 HCCL 的影响、端到端收益。

## 最终判断

A3 上 H2D 与通信冲突并不能用单纯的 CPU 抖动、Stars 队列堆积或 stream 数量过多解释；更符合当前实验现象的解释是：运行时高并发 H2D 与 HCCL/算子下发在 HCCS 共享交换路径上产生了混合流量竞争。

其中，REQ 竞争会导致动态算子下发变慢，可以通过图模式和 QoS 优先级缓解；DATA 竞争会导致 HCCL 集合通信产生的 D2D 流量被 H2D 大包阻塞，后续可优先评估拆包、限流、错峰或改变分发模式等方向。
